# Умный реестр подписок

Backend-ядро AI-платформы управления личными подписками и регулярными платежами: единый реестр обязательств, автоматический расчёт дат списаний с учётом календарных особенностей, контроль статусов и SSE-события для фронтенда.

**Стек:** Python 3.11, FastAPI, SQLAlchemy 2.0 (async) + asyncpg, PostgreSQL 16, Alembic, Docker Compose, pytest.

## 1. Как запустить проект

Нужен только Docker (Compose v2.24+):

```bash
docker compose up --build
```

Одной командой поднимаются PostgreSQL и приложение. Миграции Alembic применяются автоматически при старте контейнера (`entrypoint.sh` выполняет `alembic upgrade head` до запуска uvicorn), отдельных ручных шагов не требуется.

После старта:

- API — http://localhost:8000
- Swagger UI (OpenAPI) — http://localhost:8000/docs
- Health check — http://localhost:8000/health

Переменные окружения можно переопределить через `.env` (см. `.env.example`), по умолчанию всё работает из коробки. Если порт 8000 на хосте занят, публикуемый порт меняется переменной `APP_PORT`:

```bash
APP_PORT=8001 docker compose up --build
```

## 2. Как запустить тесты

Тесты не требуют ни Docker, ни PostgreSQL, ни сети: реальная БД подменяется на in-memory SQLite через dependency override, HTTP-вызовы идут через ASGI-транспорт без сокетов.

```bash
uv sync
uv run pytest -v
```

Покрыто (39 тестов): lazy expiry с исключением для рекуррентных обязательств, `/pay` для каждого значения `recurrence` (включая `null`), граничные случаи дат (31 января + месяц, високосный год, 29 февраля + год), попытки оплатить/отменить не-active обязательство, дубли с warning (включая кириллицу и разный регистр), окно `/upcoming` с totals и renewal_alerts, каскадное удаление и SSE-событие.

## 3. Почему lazy expiry не применяется к рекуррентным обязательствам

Перед выдачей списка (`GET /obligations`, `GET /obligations/upcoming`) сервис переводит в `expired` все `active`-записи с `next_payment_date < today` — но **только разовые** (`recurrence = null`).

Логика различия — в природе обязательства:

- **Разовое обязательство** (счёт, гарантия, страховка без продления) имеет дату истечения. Если дата прошла, а оплата не зафиксирована — обязательство объективно истекло, статус `expired` отражает реальность.
- **Рекуррентная подписка** не прекращается от того, что дата в реестре устарела. Если пользователь не нажал «Оплачено», это значит лишь, что *наш реестр* не узнал об оплате — сам сервис (Netflix, Яндекс.Плюс) продолжает действовать и спишет деньги снова. Перевести такую подписку в `expired` значило бы соврать пользователю: он перестал бы видеть её в напоминаниях, хотя списания продолжаются. Поэтому статус остаётся `active`, а устаревшая дата — сигнал «отметь оплату», а не «обязательство закончилось».

Побочный плюс «ленивой» схемы: не нужен фоновый планировщик, статусы всегда актуальны на момент чтения.

Отдельный нюанс: при **создании** обязательства с датой в прошлом статус `expired` присваивается и рекуррентным — это прямое правило ТЗ и другая ситуация. Lazy expiry имеет дело с *живой* записью, у которой устарела дата; при создании же клиент (AI-модуль) осознанно вносит постфактум уже прошедшее событие — например, распознанный старый чек, — и система фиксирует его как исторический факт, а не как действующую подписку. Если это всё же действующая подписка, клиент передаст актуальную будущую дату списания.

## 4. Граничные случаи с датами при рекуррентности

Сдвиг даты при оплате делает `dateutil.relativedelta`:

- **Сдвиг считается от текущего `next_payment_date`, а не от даты оплаты.** Если подписку от 10 мая оплатили только 6 июля, следующая дата — 10 июня, а не 6 августа: смещение при просрочках не накапливается, платёжный день сохраняется.
- **Переполнение месяца схлопывается к последнему дню.** 31 января + 1 месяц = 28 февраля (29 в високосный год), 31 октября + квартал = 31 января, 29 февраля + год = 28 февраля следующего года. Ошибки это не вызывает.
- **Осознанный компромисс:** после схлопывания день «прилипает» к новой дате — 31 января → 28 февраля → 28 марта (а не 31 марта). Чтобы всегда возвращаться к 31-му числу, нужно хранить исходный «якорный» день (`billing_anchor_day`) отдельно от расчётной даты — см. раздел про компромиссы.

## 5. Компромиссы и что бы я сделал иначе при большем времени

| Компромисс | Как сделал бы при большем времени |
|---|---|
| SSE-брокер in-process (`asyncio.Queue`): события теряются при рестарте и не работают при нескольких репликах | Redis pub/sub или Postgres LISTEN/NOTIFY; heartbeat-комментарии и `Last-Event-ID` для переподключения |
| «Прилипание» дня после короткого месяца (31 → 28 фев → 28 мар) | Хранить `billing_anchor_day` и каждый раз считать `min(anchor, дней в месяце)` |
| `today` — по времени сервера (UTC-контейнер), таймзона пользователя не учитывается | Хранить таймзону пользователя и считать «сегодня» в ней |
| Дубли ищутся точным сравнением `lower(title)` без unique-констрейнта: два конкурентных POST с одним title могут оба пройти без warning (TOCTOU) | Нормализация + триграммное сходство (`pg_trgm`); advisory lock или уникальный частичный индекс, если warning станет критичным. Констрейнт сознательно не ставлю — ТЗ требует создавать дубль, а не запрещать его |
| Lazy expiry выполняется на каждом GET отдельным UPDATE и только на читающих эндпоинтах: просроченный разовый платёж можно успеть оплатить (200), пока никто не вызвал GET | Фоновая задача по расписанию + индекс `(status, recurrence, next_payment_date)`; либо применять expiry к конкретной записи и в `/pay`/`/cancel`, сделав контракт детерминированным |
| `/upcoming` включает только `active`-обязательства — ТЗ статус-фильтр не оговаривает, но напоминать об отменённых и истёкших списаниях бессмысленно | Вынести решение продукту: возможно, отдавать параметр `include_inactive` |
| Нет аутентификации и мультипользовательности — реестр общий | `user_id` в модели + JWT/OAuth2, скоупинг всех запросов на пользователя |
| `totals` без конвертации валют (по ТЗ не требуется) | Интеграция с курсами ЦБ и сводная сумма в базовой валюте |
| Тесты на in-memory SQLite (пришлось подменить `lower()` для кириллицы) | Контрактные тесты на реальном PostgreSQL через testcontainers в CI, юнит-тесты оставить на SQLite |
| Нет пагинации `GET /obligations` | limit/offset или keyset-пагинация по `next_payment_date` |

Про конкурентность: write-эндпоинты (`/pay`, `/cancel`, `DELETE`) загружают запись с блокировкой строки (`SELECT ... FOR UPDATE`), поэтому два параллельных `/pay` по одной подписке не создадут два платежа при одном сдвиге даты — второй запрос дождётся коммита первого и получит уже не-active запись или сдвинутую дату.

## 6. Примеры запросов

Создать обязательство:

```bash
curl -s -X POST http://localhost:8000/obligations \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "Яндекс.Плюс",
    "amount": 399.00,
    "currency": "RUB",
    "category": "subscription",
    "recurrence": "monthly",
    "next_payment_date": "2027-07-31"
  }'
```

Даты в примерах — будущие относительно момента написания; при проверке позже подставьте любую будущую дату (обязательство с прошедшей датой создастся со статусом `expired` — см. следующий пример).

Создать с датой в прошлом (создастся со статусом `expired`, без ошибки):

```bash
curl -s -X POST http://localhost:8000/obligations \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "Страховка ОСАГО",
    "amount": 7800,
    "currency": "RUB",
    "category": "insurance",
    "recurrence": null,
    "next_payment_date": "2026-01-15"
  }'
```

Прислать дубль (запись создастся, в ответе появится `warning`):

```bash
curl -s -X POST http://localhost:8000/obligations \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "яндекс.плюс",
    "amount": 399.00,
    "currency": "RUB",
    "category": "subscription",
    "recurrence": "monthly",
    "next_payment_date": "2027-08-31"
  }'
# → {"obligation": {...}, "warning": "Активное обязательство с таким названием уже существует"}
```

Список с фильтрами (применяется lazy expiry, сортировка по дате):

```bash
curl -s 'http://localhost:8000/obligations'
curl -s 'http://localhost:8000/obligations?category=subscription&status=active'
```

Ближайшие списания (окно 7 дней по умолчанию):

```bash
curl -s 'http://localhost:8000/obligations/upcoming?days=30'
# → {"obligations": [...], "totals": {"RUB": 1490.0, "USD": 9.99}, "renewal_alerts": [...]}
# учитываются только active-обязательства (см. раздел «Компромиссы»)
```

Оплатить (сдвигает дату по recurrence; разовое переходит в `cancelled`):

```bash
curl -s -X POST http://localhost:8000/obligations/<id>/pay
# оплата не-active → 422
```

Отменить:

```bash
curl -s -X PATCH http://localhost:8000/obligations/<id>/cancel
# повторная отмена или отмена expired → 422 с понятным сообщением
```

Удалить (204, каскадно удаляет платежи, публикует SSE-событие):

```bash
curl -s -X DELETE http://localhost:8000/obligations/<id> -o /dev/null -w '%{http_code}\n'
```

Подписаться на SSE-события (в отдельном терминале, затем удалить любую запись):

```bash
curl -N http://localhost:8000/events
# data: {"type": "obligation_deleted", "id": "..."}
```

## Мотивация участия в проекте

**Что меня привлекает в этом проекте?**
Кейс «Финансовые напоминания» — редкий пример AI-продукта с честной, проверяемой пользой: деньги, которые перестают утекать на забытые подписки, видны сразу. Мне интересна именно связка «AI-парсинг неструктурированных данных + строгое backend-ядро»: агент может ошибаться и присылать дубли или старые чеки, а API спроектирован так, чтобы это не ломало систему (warning вместо ошибки, expired вместо отказа). Проектировать такие толерантные к недетерминированному клиенту контракты — интересная инженерная задача.

**Как я вижу свою роль в команде, которая строит AI-продукт?**
Backend-разработчик, отвечающий за детерминированное ядро: модель данных, бизнес-правила, API-контракты для AI-агентов и фронтенда, надёжность и тесты. AI-слой продукта хорош ровно настолько, насколько предсказуемо ядро, поверх которого он работает — моя зона ответственности в том, чтобы у агентов был удобный и устойчивый к их ошибкам API, а у пользователя — корректные статусы и даты.

**Сколько часов в неделю готов уделять и на какой срок?**
Готов уделять 20–30 часов в неделю на горизонте от трёх месяцев, с возможностью перехода на полную занятость.
